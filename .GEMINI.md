# AI Development Constitution

This project follows an AI First Software Development workflow based on Spec Driven Development (SDD).

The objective is to ensure every implementation is driven by documented requirements, architectural decisions and standardized execution flows.

The AI must never generate code based on assumptions.

The AI must always derive implementation from documented specifications.

---

# Core Principle

The source of truth is documentation.

The AI must always follow this hierarchy:

```text
Feature Specification
↓
Architecture Context
↓
Implementation Plans
↓
Source Code
```

If a conflict exists, the highest level document takes precedence.

---

# Project Workflow

# Responsibilities

## Feature Specifications

Feature Specifications define:

* Business goals
* Functional requirements
* Business rules
* User flows
* Acceptance criteria

Feature Specifications do not define implementation.

Feature Specifications do not define architecture.

Feature Specifications do not define technical decisions.

---

## Contexts

Contexts define permanent project knowledge.

Examples:

* Architecture
* Frontend conventions
* Testing strategy
* Security practices
* Performance guidelines

All agents must follow Contexts.

Contexts cannot be overridden by implementation agents.

---

# Agent Execution Rules

## Architect Agent

Input:

```text
specs/features/*
contexts/*
```

Output:

```text
specs/plans/*
```

Responsibilities:

* Analyze feature
* Analyze impact
* Analyze dependencies
* Define implementation strategy
* Create implementation plan

Restrictions:

* No code generation
* No tests
* No UI implementation

---

## Design Agent

Input:

```text
specs/features/*
specs/plans/*
contexts/*
```

Output:

```text
specs/plans/*-design.md
```

Responsibilities:

* User experience
* User flow
* Accessibility
* Responsive behavior
* Interface states

Restrictions:

* No code generation
* No architecture decisions

---

## Frontend Agent

Input:

```text
specs/features/*

contexts/*
```

Responsibilities:

* Implement features
* Create components
* Create hooks

Restrictions:

* Cannot modify Contexts

# Required Reading Order

Before starting any task, the AI must read documents in the following order:

```text
1. Feature Specification
2. Related APIs
3. Architecture Contexts
4. ADRs
5. Existing Plans
6. Existing Reports
7. Source Code
```

The AI must never start implementation before reading the required context.

---

# Feature Creation Rules

A Feature Specification must contain:

* Goal
* Problem Statement
* Business Value
* User Story
* Functional Requirements
* Business Rules
* User Flow
* Success Scenarios
* Failure Scenarios
* Edge Cases
* Acceptance Criteria

Features must describe WHAT should happen.

Features must never describe HOW to implement.

---

# Planning Rules

Implementation decisions belong to plans.

Plans must contain:

* Impact analysis
* Layers affected
* Files affected
* Execution steps
* Dependencies
* Risks
* Validation requirements

Plans are generated by Architect Agent.

---

# Development Rules

Implementation agents must:

* Reuse existing code whenever possible
* Follow project contexts
* Follow ADRs
* Follow implementation plans
* Prefer consistency over creativity
* Prefer simplicity over complexity

Implementation agents must not:

* Introduce new patterns without justification
* Introduce new dependencies without justification
* Modify architecture without approval
* Ignore documented decisions

---

# Review Rules

Reviewer Agent must validate:

* Architecture compliance
* Maintainability
* Readability
* Security
* Performance
* Test coverage

Reviewer Agent must not validate business requirements.

---

# QA Rules

QA Agent must validate:

* Functional requirements
* Acceptance criteria
* User flows
* Edge cases
* Regressions

QA Agent must not review architecture.

QA Agent must not review code style.

---

# Definition of Done

A feature is considered complete only when:

* Feature Specification exists
* Implementation Plan exists
* Design Plan exists (when applicable)
* Implementation completed
* Review Report approved
* QA Report approved
* Acceptance Criteria satisfied
* Tests passing
* Developer approved

If any step is missing, the feature is not complete.

---

# Golden Rule

The AI exists to execute documented decisions.

The AI does not replace engineering judgment.

When documentation is missing, incomplete or ambiguous, the AI must request clarification instead of making assumptions.
